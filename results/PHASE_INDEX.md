# Phase Index

## Purpose

This document explains how to navigate the `results/` layer of **Programmable Neurorepair** by phase rather than by isolated step folders.

The project evolved through multiple technical and conceptual blocks. Reading the results by phase is the clearest way to understand what was built, why it mattered, and how the system reached its current form.

## How to read the project by phase

The results are best understood across five major layers:

1. early engine build and state-structure derivation
2. controller refinement and boundary logic
3. biological re-entry, documentation integration, and external validation
4. intervention translation, platform framing, and final externalization
5. audience-specific packaging

These layers are cumulative.  
Each later layer depends on the earlier ones.

---

## Phase 1 — Early engine build and state-structure derivation

### Main role
This phase established the first technical basis for the project.

### What this phase built
- the earliest repair-state framing
- the first transition-oriented reading inside the remyelination wedge
- early candidate control interpretation
- the base layer for later controller logic

### Why it matters
This is the derivation layer.  
Without it, the project would not have a state map or any basis for later architecture extraction.

### How to read it
Read this phase as the **foundation**, not the final system form.

---

## Phase 2 — Controller refinement and boundary logic

### Main role
This phase transformed the project from a state map into a more operational system.

### What this phase built
- dominant repair-compatible basin interpretation
- review-zone interpretation
- boundary / frontier logic
- exception-aware logic
- rerouter family
- rescue-vs-precision tradeoff formalization
- integrated stack family
- minimal push-anchor-support architecture

### Why it matters
This is where the project stopped being only descriptive and became operational.

### How to read it
Read this phase as the **system-logic layer**.

---

## Phase 3 — Biological re-entry, documentation integration, and external validation

### Main role
This phase asked whether the internal system could be interpreted biologically and whether its core logic survived outside the original derivation environment.

### What this phase built
- biology-facing re-entry
- repo-facing documentation integration
- external validation through Dataset B
- supportive triangulation through Dataset C

### Why it matters
This is where the project stopped being only an internal sandbox.

### How to read it
Read this phase as the **biology + external-support layer**.

---

## Phase 4 — Intervention translation, platform thesis, and final externalization

### Main role
This phase translated the system into intervention-facing logic and formalized the broader platform reading.

### What this phase built
- intervention-facing program layer
- ranked validation-preparation ladder
- biological deepening of intervention logic
- platform thesis / founder layer
- final external / reviewer / founder pack
- phase closure

### Why it matters
This is where the project became externally legible as a system and not just a technical build.

### How to read it
Read this phase as the **intervention + platform + externalization layer**.

---

## Phase 5 — Audience-specific packaging

### Main role
This phase adapted the same scientific core to different external contexts without changing the underlying truth.

### What this phase built
- founder / fellowship pack
- application-tailored founder / fellowship pack
- Thiel-specific founder pack
- packaging-track separation

### Why it matters
This is where the project became ready for serious external communication.

### How to read it
Read this phase as the **presentation layer built on top of the scientific core**.
