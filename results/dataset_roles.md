# Dataset Roles

## Purpose

This document explains the role of each dataset in **Programmable Neurorepair**.

The project does not treat all datasets as interchangeable. Each dataset serves a different purpose inside the overall system.

This matters because the project now includes:

- an internal derivation layer
- an external validation layer
- a supportive triangulation layer

Without a clear dataset-role framework, it would be too easy to overstate support, flatten important differences, or confuse primary validation with secondary reinforcement.

## Why dataset roles matter

A project like this needs more than one dataset, but it also needs more than one **kind** of dataset use.

Different datasets answer different questions:

- Where is the state structure first derived?
- Where is the architecture first extracted?
- Where is external portability tested?
- Where is broader support reinforced without overclaiming?

This document defines those roles explicitly.

## Dataset A — primary derivation and state-structure sandbox

### Role
**Dataset A** is the primary derivation dataset.

It is the main internal sandbox in which the project first derived:

- the repair-state map
- the dominant repair-compatible basin
- intermediate / review-like structure
- boundary / frontier logic
- the minimal push-anchor-support architecture

### Why Dataset A matters
Dataset A is where the project first became more than a descriptive remyelination analysis.

It is the foundational dataset for:

- state-structure derivation
- transition-logic interpretation
- minimal architecture extraction
- controller and stack development

### Best reading
Dataset A should be read as:

- the **primary internal derivation environment**
- the **state-structure sandbox**
- the place where the first coherent system logic was built

### What Dataset A does not mean
Dataset A should **not** be mistaken for sufficient proof on its own.

It provides the project’s internal foundation, but not the full external support layer.

---

## Dataset B — primary external benchmark

### Role
**Dataset B** is the primary external validation benchmark.

It is the dataset used to ask whether the project’s core logic survives outside the original derivation environment.

### Why Dataset B matters
Dataset B is the strongest current external dataset because it supports a first-pass external reading of:

- architecture portability
- dominant-basin portability
- review / boundary structure portability
- integrated stack portability

### Best reading
Dataset B should be read as:

- the **main external benchmark**
- the strongest current test of whether the system is more than an internal artifact
- the main basis for first-pass external support

### Strongest current external analogues
The strongest current external readings are:

- **WT-like state** as the dominant-basin analogue
- **PLP_iCKO-like state** as the intermediate / frontier-adjacent analogue

### What Dataset B does not mean
Dataset B does **not** yet prove:

- identical geometry across datasets
- universal transfer
- broad neural state control
- completed external resolution of all ambiguity

It provides **promising first-pass external support**, not final transfer proof.

---

## Dataset C — supportive triangulation layer

### Role
**Dataset C** is a supportive triangulation dataset.

It is not the primary external benchmark. Its role is to reinforce, qualify, or complicate parts of the broader external story, especially around recovery-order structure.

### Why Dataset C matters
Dataset C strengthens the project by adding a second external perspective, but it does so in a more limited way than Dataset B.

It matters because it helps show that the project is not relying on only one external read. But it also matters that its support is more mixed and therefore must be framed honestly.

### Best reading
Dataset C should be read as:

- a **supportive triangulation layer**
- useful reinforcement
- not a co-equal benchmark
- not a replacement for Dataset B

### Current verdict
The correct current reading of Dataset C is:

- **supportive_mixed**

That means Dataset C contributes useful signal, but does not independently carry the full external story.

### What Dataset C does not mean
Dataset C should **not** be used to claim:

- that the full external story is independently proven
- that it replaces Dataset B
- that all expected recovery-order logic is cleanly reproduced
- that it provides the main external support layer

---

## Current dataset hierarchy

The current dataset hierarchy is:

### 1. Dataset A
**Primary derivation and state-structure sandbox**

### 2. Dataset B
**Primary external benchmark**

### 3. Dataset C
**Supportive mixed triangulation**

This hierarchy should remain explicit whenever the project is described seriously.

## Dataset roles across project layers

## State-structure layer
Most strongly grounded in:

- **Dataset A**

This is where the repair-state map and internal state landscape were first derived.

## Architecture layer
First derived in:

- **Dataset A**

Then externally stress-tested in:

- **Dataset B**

Supportively qualified in:

- **Dataset C**

## External validation layer
Primarily grounded in:

- **Dataset B**

Supportively reinforced by:

- **Dataset C**

## Intervention-facing layer
Derived from the architecture and state structure first built in:

- **Dataset A**

and made more credible externally through:

- **Dataset B**

with additional caution informed by:

- **Dataset C**

## Validation-preparation layer
Built on top of:

- Dataset A internal system logic
- Dataset B first-pass external support
- Dataset C supportive mixed triangulation

## Why this role separation makes the project stronger

This role separation improves the project in several ways:

- it prevents overclaiming
- it clarifies what each dataset is actually doing
- it separates internal derivation from external support
- it makes the external story more disciplined
- it prevents supportive datasets from being misread as full benchmarks

That discipline makes the project more credible, not less ambitious.

## What the current dataset-role framework supports

The project currently supports the claim that:

- Dataset A is the main derivation environment
- Dataset B is the main external benchmark
- Dataset C is supportive mixed triangulation
- external support is meaningful but not final
- dataset roles are intentionally non-interchangeable

## What the current dataset-role framework does not support

The project does **not** currently support the claim that:

- all datasets play the same role
- Dataset C replaces Dataset B
- the derivation layer alone proves external robustness
- the external layer already proves universal transfer
- dataset agreement is complete or final

## Current interpretation

The strongest current reading is:

**Dataset A builds the system, Dataset B is the main external benchmark, and Dataset C reinforces the broader story in a supportive but mixed way.**

That role separation is one of the reasons the project can now be described more seriously and more honestly.
